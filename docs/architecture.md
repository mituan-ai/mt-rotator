# 架构约束

## 领域边界

- `accounts` 管理邀请、用户和会话。
- `market` 通过 AKShare 接入新浪财经，管理 ETF 目录、24个月日线、公司行动、逐标校验和不可变数据快照。
- `strategies` 只计算版本化的全局目标，不读取用户账户，也不创建订单。
- `paper` 将全局目标与持仓批次、现金、待单和风险档位合并为建议，并负责用户委托、成交和不可变账本。
- `backtests` 从空仓自动采纳同一套建议政策，异步任务状态和结果以 PostgreSQL 为准。
- `core` 只提供权限、错误、分页、健康检查和审计等共享能力。

View 和 Serializer 只处理协议与校验，Celery Task 只领取和重试任务，事务流程位于领域 Service，策略与成交计算保持纯函数。不要增加通用 Repository、BaseService、内部事件总线或第二套任务状态。

## 数据与恢复

PostgreSQL 是唯一权威状态。Redis 只保存缓存、锁和 Celery 消息，可以重建，不纳入恢复备份。

行情修订追加保存，数据快照绑定可见修订。完整 ETF 目录均可查询，交易资格由20日平均成交额决定，建议资格还要求至少252条日线。单标的数据失败不会污染其他标的，也不会触发无依据卖出。

每名用户只有一个活跃自主账户。`PositionLot` 是 T+1、FIFO 和成本的权威记录，`AdviceSnapshot` 是账户在某个交易日的不可变建议。排行榜直接从 `NavSnapshot` 计算，不建立排行榜表。旧自动账户只读归档，不删除订单和账本。

`PaperCycleRun` 是模拟系统唯一处理水位：协调器严格按交易日推进，先处理公司行动，再处理用户开盘委托并写入净值。订单请求编号、成交键、持仓批次和账本唯一约束保证重复协调不会重复记账。

回测直接在 `BacktestRun` 上使用 15 分钟租约。旧 Worker 只有持有当前租约令牌才能提交结果；任务连续中断三次后明确失败。

## 迁移规则

线上迁移采用 expand-contract。一个普通版本只能增加表、可空字段和兼容索引。字段改名先增加新字段并双读写，后续版本切换读取，再在至少一个版本后删除旧字段。自动回退只切换应用镜像，因此旧版本必须能读取迁移后的数据库。

首次自主交易升级中的批次元数据、用户昵称唯一约束、回测租约、ETF字段和账户约束迁移已在真实 PostgreSQL 数据卷上单独验证，并以完整迁移名加入 CI allowlist。allowlist 只覆盖这五个固定文件；后续新增迁移仍由 `django-migration-linter` 拦截不兼容操作。
